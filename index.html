mysql浅谈索引

索引的常见模型

  • 哈希表是以key-value存储数据结构，只要输入key就能找到对应的value，多个key经过换算会出现同一个值，所以解决这种方法就是后面拉出个链表。这种的优缺点也比较容易看出，新增的时候简单，直接在里链表后面追加，但是不是有序的，查询就很慢。
  • 有序数组就相反，查询特别优秀，但是新增的时候，后面的数据的序号都跟着变了。
  • 二叉树的特点是每个节点都有个左儿子和有儿子，做儿子小于父节点，右儿子大于父节点，但是一颗多节点的二叉树，书树高假如20，一次查询可能就要访问20个数据库，所以为了让查询尽量少读磁盘，就使用N叉树。

InnoDB的索引模型

  • 平衡二叉树
    二叉树的左树和右树的高度差绝对值不超过1,就是平衡二叉树
  • b树
    b树可以说是二叉树的进化版，一个节点可以有多个子树。
    1. B树的关键词是：指针+关键字+地址
    2. 一个M阶b树的结点最多有M个子树

  • b+树
    b+树，是b树的一种变体，查询性能更好。m阶的b+树的特征：

    1. 有n棵子树的非叶子结点中含有n个关键字（b树是n-1个），这些关键字不保存数据，只用来索引，所有数据都保存在叶子节点（b树是每个关键字都保存数据）。
    2. 所有的叶子结点中包含了全部关键字的信息，及指向含这些关键字记录的指针，且叶子结点本身依关键字的大小自小而大顺序链接。
    3. 所有的非叶子结点可以看成是索引部分，结点中仅含其子树中的最大（或最小）关键字。
    4. 通常在b+树上有两个头指针，一个指向根结点，一个指向关键字最小的叶子结点。
    5. 同一个数字会在不同节点中重复出现，根节点的最大元素就是b+树的最大元素。

  • 平衡二叉树、b树、b+树的比较

    1. 二叉树和b树的查询次数少差不多的，但是树的高度不一样，高度h是进行磁盘IO的次数的体现，因为读写磁盘相比内存操作更加耗费时间，所以B树更合适。
    2. b树和b+树的主要区别是，b+树在非叶子结点存放关键词，不存放数据，叶子结点才存放数据。所以磁盘页能存放更多的数据，显得更“矮胖”;
    3. b+树查询必须查找到叶子节点，b树只要匹配到即可不用管元素位置，因此b+树查找更稳定(并不慢);
    4. 对于范围查找来说，b+树只需遍历叶子节点链表即可，b树却需要重复地中序遍历.

  • mysql的采用的B+树的索引模型,每个索引在InnoDB里面对应一棵B+树

索引的分类

  • 普通索引：数据库中基础索引，允许为空和重复
  • 唯一索引：允许为空单必须唯一，主键索引是一种特殊的唯一索引，不允许空也不允许重复。
  • 组合索引：多个字段组合上创建的索引，只有使用了最左边的字段才会被使用。
  • 全文索引：类型FULLTEXT，在定义索引类上支持值的全文查找。(只有MyISAM存储引擎才支持)

索引遵循最左前缀原则

  • B+树这种索引结构，可以利用索引的最左前缀来定位记录,最直观的就是假设某字段是索引,那where语句中 name like ‘小明%’会使用索引，name like ‘%小明’不会使用索引。同样，在优化索引的时候，我们也可以参考这个原则。比如 name+age的联合索引、name的索引。这个时候name的索引就可以不需要了，因为遵循最左前缀原则，name+age可以匹配name和name+age的查询条件。

设计原则

  • 索引并非越多越好，一个表中如有大量的索引，不仅占用磁盘空间，而且会影响INSERT、DELETE、UPDATE等语句的性能，因为当表中的数据更改的同时，索引也会进行调整和更新
  • 避免对经常更新的表设计过多的索引，并且索引中的列尽可能要少，而对经常用于查询的字段应该创建索引，但要避免添加不必要的字段
  • 数据量小的表最好不要使用索引，由于数据较少，查询花费的时间可能比遍历索引时间还要短，索引可能不会产生优化效果
  • 在条件表达式中经常用到的不同值较多的列上建立索引，在不同值较少的列上不要建立索引，比如性别字段只有男和女，就没必要建立索引。如果建立索引不但不会提高查询效率，反而会严重降低更新速度
  • 当唯一性是某种数据本身的特征时，指定唯一索引。使用唯一索引需能确保定义的列的数据完整性，以提高查询速度
  • 在频繁排序或分组（即group by或order by操作）的列上建立索引，如果待排序的列有多个，可以在这些列上建立组合索引

项目拆分多个模块

简单认识下pom文件各个标签

模块拆分

现在项目中一般使用maven进行模块管理，各个模块都有自己的pom文件，pom文件包含了模块之间的依赖和继承关系。
在进行模块划分管理时候，一般都会有一个父模块，父模块的packing是pom，可以给各种需要引用的依赖定义版本，这样子模块在引用依赖的时候就不用指定版本号，划分的时候，可以将通用的部分抽离出来，进行重复使用。可以通过modules将子模块引用进来，在build父项目时候，会根据子模块的相互依赖关系整理出build顺序，依次build.

  1. 根目录pom文件作为父文件，定义了相关的依赖的引用版本，以及包含的模块。
  2. 将公用的类以及必须的实现提取到base里面，在其pom文件引进公用的依赖。
  3. 其他模块在pom文件引入base依赖，再根据自己各自的需求引进依赖。

特殊需求的发号器

需求:大多数业务都要求生成编码来唯一识别每一条数据,根据需求有以下几种情况.

  1. 固定长度补0的自增长编号
  2. 不补0的自增长编号
  3. 时间戳+第一种情况
  4. 时间戳+第二种情况
  5. 前缀+第一种情况
  6. 前缀+第二种情况
  7. 前缀+时间戳+第一种情况
  8. 前缀+时间戳+第二种情况
    能够根据业务不同设定不同的code,code保证唯一性。

设计

根据需求,考虑到需求多种策略来执行,以及高可用高性能,设计了以下表:

  • code用于标示是什么业务的编码生成规则
  • init_value是初始化的值
  • current_value是当前的值
  • num_length是编号的自增长的长度
  • prefix是知前缀
  • cache_frequency是更新缓存的频率,假如是5则获取5次进行 一次数据库和redis的更新(用于提高请求的效率)
  • step默认是1,指递增的步伐
  • strategy指需求对应下的几种算法

实现

  • 对于需要多种不同算法来实现不同的业务,采用策略模式设计模式,
    根据context类获取对应的策略来进行生成编号。此处的策略模式假如增加的时候,需要增加对应的策略以及修改context,为了更符合开闭原则,可以优化成用反射获取等。
  • 数据的长度表示最大的值是10的n次方-1，假如是时间戳的算法，表示每天支持最多生成的编码是10的n次方-1
  • 为保证数据的唯一性以及原子性,数据在获取下一个的时候采用redis的原子计算器来保证原子性。假设内存丢失的情况,则获取redis缓存的数据key更新到内存,假如redis数据也丢失,则获取数据库的数据取下次更新的最大值作为当前值来进行生成,避免生成重复数据.

导出大数据量excel

2019-04-01

背景:随着报表数据的增多,全表导出造成OOM,需要对现有系统的导出excel功能做进一步优化.

现状:通过查询出所有数据data,再查询出对应的需要转换的字段的字典,对data进行for循环处理数据,采用的是poi的XSSFworkbook,再对data数据for循环处理写到workbook。

优化:

  • 对data进行了太多次for循环处理,而且一次性加载太多数据容易造成OOM,mysql本身支持流式查询,上网查了下mybatis的流式处理可以使用ResultHandler来进行实现。多次的for循环处理数据可以直接在实现handleResult()方法中进行优化处理。
  • poi的XSSFworkbook支持的数据量不大,网上支持大数据的大都通过SXSSFworkbook,在new的时候指定内存保留的数据条数,可以避免造成OOM,而且也可以把写到workbook的操作也写到handleResult中。

总结:从原先的三次for循环变成一次for循环,采用SXSSFworkbook支持了大数据量的导出,避免的大数据量导出过程的OOM,通过减少对数据的多次处理,大大减少了导出时间。

核心代码

handleResult方法是实现了ResultHandler接口的方法,相当于在查询data的时候会进行调用处理,DataConverter是数据处理类,在此基础可以有UserDataConverter;DataWriter是数据写入类,在此基础可以有ExcelDataWriter.

MySQL的查询过程

2018-09-01

MySQL分为Server层和存储引擎层两部分

  • Server层包括连接器、查询缓存（一般没用）、分析器、优化器、执行器，很多功能都在这一层上面实现，mysql的核心功能、内置函数、跨引擎的功能等。
  • 存储引擎主要负责数据的存储和提取，主要有InnoDB、MyISAM、Memory等多种引擎。最常用的是InnoDB。

查询语句的查询过程

如:select * from user where id = 10;

  1. 先连接数据库，这时候连接器负责与客户端建立连接、获取权限、维持和管理连接。
  2. 连接建立完成后，会先到查询缓存看看，之前是否执行过这条语句，执行语句和结果是以key-value的形式缓存在内存中。（因为缓存往往利大于弊，所以不建议开启，8.0版本之后整块缓存功能已经删除。）
  3. 第三步是分析器对SQL进行解析，比如对select解析出是查询语句id解析出是列名。一般sql写错的时候会报错误，而且会第一个出现错误的位置。
  4. 经过分析器之后，在开始执行sql之前，会经过优化器，优化器是在表里面有多个索引的时候，决定使用哪个索引、或者是夺标关联查询join的时候，决定各个表的连接顺序。
  5. 优化器之后，语句的执行方案就出来了，就进入执行器阶段。开始执行的时候，先判断你是否对表有没有执行查询的权限，如果没有就返回没有权限的错误。如果有权限，就打开表继续执行，执行器会根据表的引擎定义，去使用这个引擎提供的接口。
  • 如果没索引则取第一行判断是否id是否等于10，如果不是则循环吓一跳，是则存入结果集然后执行下一行，直到表的最后一行。
  • 对于有索引的，执行逻辑类似，取满足条件的第一行的接口，之后循环。
  • 使用过explain的就看到过一个rows_examined的字段，表示执行过程扫描了多少行，这个值就是执行器每次调用引擎获取数据行的时候累加的。

算法之插入排序

2018-07-20

  1. 思想：插入排序首先是将第一位数看成是已经排序号的数列，第二位与之比较插入到合适的位置，这时候一二位就变成了已经排序的数组，接着第三位拿来和第二位比较，依次类推，直到排序完成。

  2. 时间复杂度分析，最坏情况下是全部逆序，插入第一个需要考虑一个元素，插入第二个需要考虑两个元素，最坏情况下是1+2+……+(n-1）,等差数列求和就是n^2 /2;O( n^2 );最好情况是O(n),就是只考虑一个元素;平均是O( n^2 );空间复杂度是O(1)。

算法之冒泡排序

2018-07-14

1.冒泡排序：简而言之，从最底部开始比较相邻的两个数的值大小，如果大的在底部，则交换。大的永远得在上面，这样最大的永远在上面，以此类推。

2.冒泡排序的最差时间分析是O( n^2 ),平均时间复杂度O( n^2 ),稳定，空间复杂度是O(1)。
